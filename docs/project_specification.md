# JG Business 프로젝트 사양서

## 1. 문서 목적

본 문서는 `jg_business` 프로젝트의 현재 구현 상태를 기준으로 제품 범위, 기능 요구사항, 화면 구조, 데이터 구조, 외부 연동, 비기능 요구사항을 정리한 사양서다.

- 기준일: 2026-03-19
- 기준 소스: 현재 저장소의 Flutter 앱 및 `docs/`, `firebase/` 문서
- 작성 원칙: 기획 의도와 실제 구현을 구분한다.

## 2. 프로젝트 개요

JG Business는 영업 담당자가 Google Calendar 기반 일정, 면담 기록, 고객 정보, 명함 입력, 알림을 한 앱에서 관리하도록 설계된 Flutter 멀티플랫폼 업무 앱이다.

핵심 방향은 다음과 같다.

- 캘린더를 단순 조회 도구가 아니라 영업 액션의 시작점으로 사용한다.
- 일정 이후 반드시 면담 기록과 후속 조치로 연결한다.
- 고객 정보와 면담 기록을 Firestore에 축적하고, 필요 시 Google Sheets로 동기화한다.
- 모바일 사용을 우선하되 웹/데스크톱에서도 동일 코드베이스를 유지한다.

## 3. 제품 목표

- 오늘 처리해야 할 영업 일정을 빠르게 파악할 수 있어야 한다.
- 일정별 면담 기록 작성 여부를 추적할 수 있어야 한다.
- 면담 종료 후 기록 누락을 줄이기 위해 로컬 알림을 제공해야 한다.
- 고객 정보와 최근 접점을 한 화면에서 확인할 수 있어야 한다.
- 명함 정보는 초기에는 수동 OCR 텍스트 검수 방식으로라도 고객 등록까지 이어져야 한다.
- 면담 기록은 Google Sheets에 1차 동기화할 수 있어야 한다.

## 4. 대상 사용자

- Google Calendar를 사용하는 영업 담당자
- 고객 방문, 면담, 후속 조치 기록이 필요한 B2B 세일즈 담당자
- 일본 로케일/시간대 기준으로 일정을 운영하는 사용자

## 5. 지원 플랫폼

- Flutter 기반 멀티플랫폼
- 확인된 타깃: iOS, Android, Web, macOS, Windows, Linux
- 현재 UX 기준 우선순위
  - 모바일 우선
  - iPad/대화면 분할 레이아웃 일부 고려
  - 웹은 Google 로그인과 캘린더 사용 보조 채널 성격

## 6. 기술 스택

- 앱 프레임워크: Flutter
- 상태관리/라우팅/DI: GetX
- 백엔드: Firebase Core, Cloud Firestore
- 인증: Google Sign-In
- 외부 API: Google Calendar API, Google Sheets API
- 네트워크: Dio
- 로컬 저장소: SharedPreferences
- 로컬 알림: flutter_local_notifications
- 위치: geolocator, geocoding
- 일정 UI: table_calendar
- 국제화/포맷: intl, flutter_localizations

## 7. 앱 기본 동작

### 7.1 초기화

앱 시작 시 다음을 수행한다.

- 일본어 로케일 데이터 초기화
- 타임존을 `Asia/Tokyo`로 고정
- Firebase 초기화
- 테마 설정 로드

### 7.2 첫 실행 플로우

- 스플래시 화면에서 알림 서비스 초기화 및 권한 요청
- 로컬 저장소의 온보딩 완료 여부 확인
- 미완료 시 온보딩으로 이동
- 완료 시 메인 화면으로 이동

### 7.3 온보딩

- 3페이지 소개형 온보딩 제공
- 완료 또는 스킵 시 온보딩 완료 플래그 저장
- 이후 Calendar 연결 소개 화면으로 진입

### 7.4 캘린더 연결 정책

- 최초 진입 시 Google 로그인은 강제하지 않는다.
- 사용자는 홈 진입 후 필요 시 Google Calendar를 연결할 수 있다.
- 캘린더 미연결 상태에서도 앱 일부 화면은 접근 가능하다.

## 9. 기능 사양

### 9.1 인증

#### 목적

- Google 계정 기반 세션을 관리한다.
- Calendar/Sheets 접근 권한을 확보한다.

#### 현재 구현

- Google Sign-In 기반 로그인 상태 관리
- 현재 사용자 이메일, 표시 이름, 사용자 ID 접근
- 세션 변경 감지
- 재인증 수행
- 로그아웃 수행

#### 비고

- 문서에는 Okta SSO가 언급되어 있으나 현재 코드 기준으로는 구현 범위에 포함되지 않는다.

### 9.2 홈

#### 목적

- 오늘의 영업 흐름을 빠르게 요약한다.

#### 현재 구현

- 요약 카드
- 다음 면담 카드
- 기록 대기 카드
- 최근 기록 카드

#### 데이터 출처

- Google Calendar 이벤트
- Firestore의 `meeting_status`
- Firestore의 `meeting_records`

#### 주요 규칙

- 진행 중 면담이 있으면 우선 노출
- 없으면 가장 가까운 다음 일정을 기준으로 표시
- 기록 완료된 일정은 후속 기록 알림 대상에서 제외

### 9.3 캘린더

#### 목적

- Google Calendar 일정 조회 및 CRUD를 제공한다.

#### 현재 구현

- Google Calendar 연결
- 현재 시점 기준 약 6개월 전부터 6개월 후까지 일정 조회
- 주간 기본 보기 + 확장형 월간 캘린더 보기
- 날짜별 일정 목록 조회
- 오늘 일정 / 향후 일정 섹션 제공
- 일정 생성
- 일정 수정
- 일정 삭제
- 일정 상세 화면 이동

#### 화면 규칙

- 미연결 상태에서는 연결 유도 화면을 표시한다.
- 연결 상태에서만 일정 추가 FAB를 노출한다.
- 월/주 전환은 접이식 패널로 처리한다.

#### 데이터 원칙

- 일정 CRUD의 source of truth는 Google Calendar다.
- Firestore는 일정 자체가 아닌 영업 운영 메타데이터를 저장한다.

### 9.4 면담 기록

#### 목적

- 일정에 대응되는 면담 결과를 기록하고 후속 액션까지 남긴다.

#### 현재 구현

- 일정 기반 면담 기록 화면 진입
- 동일 Google 이벤트 ID 기준 기존 기록 조회
- 신규 저장 및 수정 모드 지원
- 입력 항목
  - 회사명
  - 담당자명
  - 요약
  - 상세 메모
  - 다음 액션
- 저장 시 이벤트 주소를 좌표로 변환 시도
- 저장 시 고객 데이터 upsert
- 저장 시 `meeting_status.recordStatus`를 `completed`로 반영
- 기록 삭제
- 삭제 시 상태를 기록 미완료 상태로 복원

#### 저장 규칙

- `summary`는 필수 입력이다.
- 기록 ID는 기본적으로 `{userId}_{googleEventId}` 형식을 사용한다.
- 저장 후 캘린더 상태를 재조회해 홈/캘린더 UI를 동기화한다.

#### 현재 제한

- 다음 액션 기한(`nextActionDueAt`)은 아직 UI/저장 로직에서 사용하지 않는다.
- AI 요약은 README에 방향성만 있고 현재 구현되지 않았다.

### 9.5 면담 상태 관리

#### 목적

- 일정별 기록 필요 여부와 알림 관련 상태를 앱 소유 데이터로 관리한다.

#### 현재 구현

- 캘린더 이벤트 목록을 기준으로 `meeting_status` 동기화
- 기록 완료 여부 추적
- 홈의 기록 대기 목록 산출
- 완료된 일정은 면담 후 알림 해제 대상에 포함

#### 확장 예정

- 도착/이탈 기반 상태
- 후속 조치 상태
- 알림 발송 이력

### 9.6 고객 관리

#### 목적

- 고객사 및 담당자 정보를 면담 기록과 연결해 관리한다.

#### 현재 구현

- 사용자 기준 고객 목록 조회
- 회사명/담당자명 검색
- 고객 상세 조회
- 고객별 연결된 면담 기록 목록 조회
- 고객 정보 수정
  - 회사명
  - 담당자명
  - 전화번호
  - 이메일
  - 메모
- 최근 접점 및 연결 면담 수 표시

#### 레이아웃

- 대화면에서는 목록/상세 분할 레이아웃
- 모바일에서는 목록 후 상세 페이지 이동

#### 연계 규칙

- 면담 기록 저장 시 선택한 고객 또는 입력한 고객명 기준으로 고객 데이터 upsert
- 명함 등록 저장 시 고객 데이터 upsert

### 9.7 명함 등록

#### 목적

- 명함 정보를 고객 정보로 빠르게 등록한다.

#### 현재 구현

- OCR 원문 텍스트 수동 입력
- 간단한 규칙 기반 초안 추출
  - 회사명 후보
  - 담당자명 후보
  - 전화번호
  - 이메일
- 사용자가 추출값 검수 후 저장
- 저장 시 `business_cards`와 `clients` 동시 반영

#### 현재 제한

- 카메라 촬영 없음
- 이미지 업로드 없음
- 실제 OCR 엔진 연동 없음
- 추출 로직은 정규식/문자열 규칙 기반의 임시 구현

### 9.8 알림

#### 목적

- 일정 전후 시점에 기록/준비를 유도한다.

#### 현재 구현

- 로컬 알림 초기화
- 플랫폼별 알림 권한 요청
- 알림 허용 여부 조회
- 캘린더 일정 기반 알림 재동기화
- 일정 시작 전 알림
- 일정 종료 후 2회 리마인드 알림
- 연결 해제되거나 사라진 일정의 알림 정리

#### 규칙

- 종일 일정은 알림 대상에서 제외
- 기록 완료된 일정은 종료 후 알림을 취소
- macOS에서는 일부 일정 알림 스케줄링을 생략

#### 향후 확장

- 위치 기반 알림
- 푸시 알림
- 알림 탭/이력 관리

### 9.9 위치 권한 및 주소 해석

#### 목적

- 향후 위치 기반 면담 트리거를 위한 기반을 제공한다.

#### 현재 구현

- 위치 서비스 활성화 여부 확인
- 위치 권한 상태 확인
- 앱 사용 중 권한 요청
- 항상 허용 권한 요청
- OS 설정 화면 이동
- 일정 주소 문자열을 좌표로 해석
- 일본 주소 특성을 고려한 주소 단순화 후보 재시도

#### 현재 상태 판단

- 위치 권한 관리 UI는 일부 구현됨
- 지오펜스 기반 도착/이탈 트리거는 아직 완성되지 않음

### 9.10 Google Sheets 동기화

#### 목적

- 면담 기록을 스프레드시트로 내보내 업무 공유를 단순화한다.

#### 현재 구현

- 면담 기록 저장 직후 Sheets 동기화 시도
- 시트 헤더 자동 보장
- `recordId` 기준 기존 행 탐색 후 update 또는 append
- 동기화 상태 저장
  - `pending`
  - `syncing`
  - `synced`
  - `failed`
- 실패 코드 저장

#### 동기화 컬럼

- 예정 제목
- 회사명
- 담당자명
- 시작日時
- 종료日時
- 장소
- 요약
- 상세 메모
- 다음 액션
- 사용자 ID
- Google 이벤트 ID
- 레코드 ID

#### 현재 제한

- 단일 스프레드시트/시트명 기준 동작
- 배치 동기화 없음
- 백그라운드 재시도 없음

### 9.11 활동 탭

#### 현재 구현

#### 기획 방향

- 통화 요약

- 동기화 실패 항목
- 통합 타임라인

### 9.12 기타 탭

#### 현재 구현

- 캘린더 재연결
- 로그아웃
- 알림 권한 상태 확인 및 재요청
- 예약된 캘린더 리마인더 개수 확인

## 10. 데이터 저장 사양

### 10.1 원칙

- Google Calendar: 일정 데이터의 원본
- Firestore: 영업 운영 메타데이터의 원본
- SharedPreferences: 로컬 앱 상태 저장

### 10.2 Firestore 컬렉션

현재 문서 및 구현 기준 주요 컬렉션은 다음과 같다.

- `users`
- `meeting_records`
- `meeting_status`
- `clients`
- `business_cards`

### 10.3 `meeting_records`

주요 필드:

- 사용자 ID
- Google 이벤트 ID
- 캘린더 ID
- 일정 제목
- 회사명
- 담당자명
- 시작/종료 시각
- 장소명
- 위도/경도
- 요약
- 메모
- 다음 액션
- 상태
- Sheets 동기화 상태
- 생성/수정 시각

### 10.4 `meeting_status`

주요 필드:

- 사용자 ID
- Google 이벤트 ID
- 캘린더 ID
- 시작/종료 시각
- 장소명
- 위치 좌표
- 기록 상태
- 리마인드 상태
- 후속 상태
- 마지막 알림 시각
- 생성/수정 시각

### 10.5 `clients`

주요 필드:

- 사용자 ID
- 회사명

- 메모

## 11. 외부 연동 사양

### 11.1 Google Calendar

- 사용 목적: 일정 조회/생성/수정/삭제
- 권한 범위: Calendar 접근 권한 필요
- 동작 범위: `primary` 캘린더 기준

### 11.2 Google Sheets

- 사용 목적: 면담 기록 행 단위 업서트
- 권한 범위: Sheets 접근 권한 필요
- 대상: 환경값으로 지정된 Spreadsheet ID와 Sheet Name

### 11.3 Firebase

- 사용 목적: 앱 데이터 저장
- 구성: Firebase Core, Firestore

### 11.4 위치/지오코딩

- geolocator: 권한 및 설정 상태 조회
- geocoding: 주소를 위경도로 변환

## 12. 환경 설정 값

런타임 환경값은 다음을 사용한다.

- `GOOGLE_WEB_CLIENT_ID`
- `GOOGLE_SHEETS_SPREADSHEET_ID`
- `GOOGLE_SHEETS_SHEET_NAME`

주의사항:

- 현재 코드에는 기본값이 포함되어 있다.
- 운영 환경에서는 빌드/배포 체계에서 별도 관리하는 것이 바람직하다.

## 13. 비기능 요구사항

### 13.1 로케일 및 시간대

- 기본 로케일은 `ja_JP`
- 지원 로케일은 일본어, 영어
- 타임존은 `Asia/Tokyo`

### 13.2 반응형 UI

- 모바일 우선 설계
- 고객 화면은 대화면 분할 레이아웃 지원
- 주요 콘텐츠 영역 폭 제한으로 태블릿/데스크톱 가독성 확보

### 13.3 상태 동기화

- 인증 세션 변경 시 캘린더 재조회
- 면담 기록 저장/삭제 후 홈/캘린더 상태 재동기화
- 캘린더 조회 후 알림 재동기화

### 13.4 오류 허용 전략

- 일부 외부 API 실패 시 앱 전체를 중단하지 않고 빈 상태 또는 실패 상태로 처리
- Sheets 동기화 실패는 면담 기록 저장 실패로 간주하지 않는다.
